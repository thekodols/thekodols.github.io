Tools for front-end webapp development on Node.js

The world of web development has been maturing rapidly over the last couple years, with a deluge of new tools to help you build more complex sites faster. There has been a lot of activity, particularly in the JavaScript community, in large part thanks to the rise of Node.js and npm, allowing for a much more diverse range of tools to be released, making things like better integrated package managers, build tools, and frameworks possible.

These tools can greatly improve your development speed, but because of how fast this area has been changing, and how active the community has been, it can be hard to wrap your head around all the different types of tools, and the differences between your options. To help you keep track of it all, I've created an overview of different tools for frontend development, with an explanation of what problem they solve, and what options there are.

Workflow Wrappers

Keeping track of every single tool you'll need, even within the scope of a single project, is not an easy task. Doing it by hand would be near impossible. To address this issue there are workflow wrappers.

Workflow wrappers help manage all of the tools you'll need for front-end webapp development, like build systems, package managers, frameworks, etc., allowing you to get started quickly and keep sane throughout.

All of these wrappers are somewhat opinionated, meaning they enforce (or strongly suggest) a certain set of tools. Some do it more, some do it less. And if you're just starting out, it's a good idea to stick with their suggested workflows and see what parts to switch out for others when you've reached a certain understanding of how these tools interoperate.

Build Systems

Build systems (a.k.a. task runners) provide a standardized way to execute the commands you need to build and serve your site. They take care of things like running a development server, automatic linting, compiling, concatenating files, minifying, running tests, deploying, etc. They automate your workflow so you don't have to do repetitive tasks each time you make an edit.

Build tools allow for rapid iteration and will save vast amounts of time on any project that requires writing anything past vanilla html, css or js.

View in fullscreen: A comparison of Node.js build systems / task runners

Here, a watch task, that either comes as a plugin or is already baked into these build tools, will run a set number of tasks the moment a file is modified. Couple that with a tool that live-updates & synchronizes your browser and you'll be able to see the implemented changes as soon as you save a file.

Note that build systems don't excuse you from not knowing how things operate under the hood.

Package management

To help you install, upgrade and manage external libraries you can use front-end package managers.

The "standard" way of managing front-end JavaScript packages is to just load up a bunch files via script tags. This has many problems such as creating multiple new requests, being hard to manage across the site, hard to manage versions and hard to manage dependency load order. A front end package manager solves some or all of these problems.

View in fullscreen: A comparison of front-end package managers

Additionally, since JavaScript doesn't come with a way to modularize code (yet) you might want to look into JavaScript module loaders that will take care of dependency management and concurrency.

Boilerplate

Boilerplate is any code that's included in the beginning of a project with little or no alteration. It lays the foundation of what you're building. Taking care of common design principles, like responsive layout, folder structure, etc so you don't have to build them from scratch each time you start a new project. A good boilerplate will follow years of best practices.

There are a few things to keep in mind. Since it's any foundation code, it can include support for authoring abstractions, specific tools or even complete frameworks.

If you want to pick out all your tools by yourself a safe bet is HTML5 Boilerplate, but you might find other options more compelling depending on what best fits for a particular project.

Authoring Abstractions

As far as the browser is concerned there are three languages - HTML for structure, CSS for layout and style, and JS for anything dynamic. For various reasons, but mainly ease of use and speed, you might want to consider writing in languages that compile down to each of these three languages instead of writing them directly.

JavaScript

JavaScript might be one of the most hated languages around, but we're mostly stuck with it if we intend to write a modern webapp. So to work around some of the questionable parts of JS here's a list of languages that compile down to JavaScript.

There are many different compiled languages to match your personal preferences. Depending on whether you want better type safety, or a syntax more conductive to functional programming, or better object oriented programming support, you'll want to find the one that you work best with.

CSS

Writing CSS has always felt hackish. For example, there are no variables in CSS so deciding to change your site's color scheme just slightly involves having to look for each instance of a particular color for each element. Not the most efficient use of your time.

To avoid many of the headaches with CSS, there are pre-processors, that greatly increase ease of developing the look of your site. They add things like variables, mixins and functions among other features that make CSS more maintainable and extendable.

HTML

For HTML, there are templating engines. Templating engines take in a template file and a data set, interpolates them, outputting HTML. Not only do most templating engines provide a shorthand, human-readable way of writing HTML, they also make sure that the markup is semantic and accessibility compliant.

A good idea before choosing a templating engine is to figure out how much logic you want in them.

View in fullscreen: A comparison of JavaScript templating engines

There's also Zen Coding and Emmet, editor plugins, that allows for much quicker writing of vanilla HTML, if you decide against using a templating engine.

Frameworks

Frameworks are a layer of abstraction intended to help you build your application quickly. It is one of the bigger commitments you can make, but also has the most impact on development speed.

There are Node.js frameworks for back-end development, that handle things like routing and authentication, and client side frameworks for front-end development, for working with how the requested data is displayed and managing the DOM and user interactions.

View in fullscreen: A comparison of JS front-end frameworks

There's also a list of CSS frameworks to choose from. They provide a consistent way of structuring classes and applying styles.

As with all tools, but frameworks especially, working without one first will help you understand what they add to your workflow. A good idea is trying out a few of them before picking one that feels comfortable to you.

Development Environments

Choosing the right text editor, exploring its possibilities and setting it up correctly is essential. Make sure you have good plugins for common actions, that take care of things like linting, syntax highlighting, and autocomplete, but that's more or less self-explanatory.

Now, you don't necessarily have to choose a text editor - you can go with an IDE, or even a cloud IDE - but for most front-end developers IDEs are overkill.
